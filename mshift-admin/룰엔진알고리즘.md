# MoneyShift 룰엔진 알고리즘 분석 보고서

## 개요

MoneyShift 프로젝트에는 현재 **4가지 주요 룰엔진 알고리즘**이 구현되어 있으며, 각각 다른 용도와 성능 특성을 가지고 있습니다.

---

## 1. 정규식 기반 룰 엔진 (RegexRuleEngine)

### 📁 위치
- `lib/services/regex-rule-engine.ts`
- `__tests__/services/regex-rule-engine.test.ts`

### 🔧 알고리즘 특징
- **하드코딩된 정규식 패턴** 사용
- **신뢰도 기반 우선순위** 매칭
- **카테고리별 특화 정규화** 로직

### ⚙️ 동작 방식
```
입력: "지에스칼텍스(상)주석계"
↓
1. 모든 규칙에 대해 정규식 매칭 수행
2. 매칭된 규칙 중 최고 신뢰도 선택
3. 카테고리별 정규화 함수 적용
↓
출력: {
  matched: true,
  category: "주유소",
  pattern: "(상)주",
  confidence: 0.95,
  normalizedName: "GS칼텍스"
}
```

### 📊 성능 특성
- **매칭 속도**: 매우 빠름 (메모리 내 처리)
- **정확도**: 높음 (수동 튜닝된 패턴)
- **유지보수**: 어려움 (코드 수정 필요)

### 🎯 지원 카테고리
1. **주유소** (신뢰도: 0.8~0.95)
   - `(상)주`, `(하)주` 패턴: 0.95
   - 셀프 주유소: 0.9
   - 브랜드명 패턴: 0.85
   - 일반 주유소: 0.8

2. **편의점** (신뢰도: 0.95)
   - GS25, CU, 세븐일레븐, 이마트24, 미니스톱

3. **카센터** (신뢰도: 0.85~0.9)
   - 카센터 키워드: 0.9
   - 자동차 서비스센터: 0.88
   - 정비소: 0.85

4. **온라인서비스** (신뢰도: 0.95)
   - Google, Amazon, Microsoft, GoDaddy

## 2. CSV 정규식 룰 엔진 (CsvRegexRuleEngine)

### 📁 위치
- `lib/services/csv-regex-rule-engine.ts`
- `data/regex-rules.csv`
- `__tests__/services/csv-regex-rule-engine.test.ts`

### 🔧 알고리즘 특징
- **외부 CSV 파일**에서 정규식 패턴 로드
- **동적 규칙 리로딩** 지원
- **싱글톤 패턴** 구현

### ⚙️ CSV 파일 구조
```csv
pattern,category,confidence,description,normalizer_type,enabled
"GS25",편의점,0.95,"GS25 편의점 패턴",convenience_store,true
"(.+?)카센터",카센터,0.9,"카센터 키워드 패턴",car_center,true
"(GOOGLE|구글)",온라인서비스,0.95,"Google 온라인서비스 패턴",online_service,true
```

### 📊 성능 특성
- **매칭 속도**: 빠름
- **유연성**: 높음 (CSV 수정만으로 룰 변경)
- **관리 편의성**: 우수함

### 🔄 동작 흐름
```
1. CSV 파일 파싱 및 정규식 컴파일
2. enabled=true인 규칙만 로드
3. 신뢰도 순으로 정렬
4. 매칭 시 최고 신뢰도 규칙 반환
5. 카테고리별 정규화 함수 적용
```

## 3. 키워드 기반 룰 엔진 (DB Rule Engine)

### 📁 위치
- `app/api/rule-engine/search/route.ts`
- `components/combined-rules-management.tsx`

### 🔧 알고리즘 특징
- **PostgreSQL 데이터베이스** 저장 룰
- **단순 키워드 포함 매칭**
- **사용 통계 수집** 및 학습

### ⚙️ 동작 방식
```
입력: "스타벅스 강남점"
↓
1. 입력 텍스트를 소문자로 정규화
2. DB에서 활성 룰 전체 조회 (신뢰도 내림차순)
3. 키워드가 포함된 룰 필터링
4. 최고 신뢰도 룰 선택
5. 사용 횟수 증가 (usage_count++)
↓
출력: {
  success: true,
  matches: [{
    id: 1,
    keyword: "스타벅스",
    confidence: 85,
    tag: "#접대비",
    account: "접대비"
  }]
}
```

### 📊 성능 특성
- **매칭 속도**: 보통 (DB 쿼리 필요)
- **학습 능력**: 있음 (사용 통계, 피드백)
- **확장성**: 우수함

### 🎯 DB 스키마
```sql
RuleEngine {
  id: number
  keyword: string          -- 매칭할 키워드
  confidence: number       -- 신뢰도 (0-100)
  primaryTag: string       -- 1순위 태그
  primaryAccount: string   -- 1순위 계정과목
  secondaryTag: string     -- 2순위 태그
  secondaryAccount: string -- 2순위 계정과목
  usageCount: number       -- 사용 횟수
  positiveCount: number    -- 긍정 피드백 수
  negativeCount: number    -- 부정 피드백 수
  isActive: boolean        -- 활성 상태
}
```

## 4. 하이브리드 룰 엔진 (Rule DB + LLM)

### 📁 위치
- `project-design/runleendgine.md`
- `components/rule-engine-content.tsx`

### 🔧 알고리즘 특징
- **계층적 처리** 구조
- **비용 최적화** 설계
- **자체 학습** 루프

### ⚙️ 5단계 처리 흐름

#### 1단계: 전처리 및 정규화
```
입력: "(주)스타벅스커피코리아 강남역점 9,800원"
↓
- 거래처명/금액 분리
- 불필요한 단어 제거 (법인형태, 지점명 등)
↓
출력: "스타벅스커피"
```

#### 2단계: 룰 DB 우선 검색
```
정제된 거래처명 → DB 키워드 매칭
성공 시: 3단계로 이동
실패 시: 4단계(LLM)로 이동
```

#### 3단계: 규칙 적용 및 사용자 상호작용
```
신뢰도 >= 90%: 자동 적용
신뢰도 70-89%: 사용자 선택 요청
신뢰도 < 70%: LLM 호출
```

#### 4단계: LLM 추론
```
입력: 원본 거래 문자열
↓ 
LLM (Gemini/GPT) 호출
↓
태그/계정과목 추천 + 신뢰도
```

#### 5단계: 학습 루프
```
사용자 피드백 수집
↓
- 긍정: 신뢰도 +1
- 부정: 신뢰도 -2, 순서 교체
- LLM 결과 확정: 신규 룰 후보 생성
```

## 성능 비교 및 권장사항

### 📊 **처리 속도 비교**
1. **정규식 엔진**: ~0.1ms (가장 빠름)
2. **CSV 정규식**: ~0.5ms (파일 I/O 포함)
3. **키워드 DB**: ~5-20ms (DB 쿼리)
4. **LLM 호출**: ~500-2000ms (가장 느림)

### 💰 **비용 비교**
1. **정규식**: 거의 무료
2. **키워드 DB**: DB 비용 (매우 저렴)
3. **LLM**: 호출당 비용 (상대적으로 비쌈)

### 🎯 **최종 권장 아키텍처**

```
[입력 거래 문자열]
        ↓
[1단계: 고신뢰도 정규식 엔진] ← 확실한 패턴 (주유소, 편의점 등)
        ↓ (매칭 실패 시)
[2단계: CSV 동적 정규식 엔진] ← 관리자가 추가한 패턴
        ↓ (매칭 실패 시)  
[3단계: DB 키워드 엔진] ← 학습된 키워드 룰
        ↓ (매칭 실패 시)
[4단계: LLM 추론] ← 완전히 새로운 패턴
        ↓
[5단계: 학습 루프] ← 피드백 반영 및 룰 생성
```

이 구조를 통해 **속도**, **정확도**, **비용 효율성**을 모두 달성할 수 있습니다.

## 개선 제안사항

1. **정규식 엔진 확장**: 음식점, 병원, 약국 등 추가 카테고리
2. **CSV 관리 도구**: 웹 UI를 통한 정규식 패턴 관리
3. **성능 모니터링**: 각 단계별 처리 시간 및 성공률 추적
4. **A/B 테스트**: 다양한 알고리즘 조합의 효과 측정
